# Техническое задание: «Паспорт автоматизации бизнеса»

## 1. Цель проекта

Создать современный интерактивный веб-сервис для предварительной диагностики бизнес-процесса.

Сервис должен помочь предпринимателю, руководителю, эксперту или самозанятому понять:

* какие ручные и повторяющиеся действия можно автоматизировать;
* сколько времени уходит на текущий процесс;
* какой эффект может дать автоматизация;
* какой формат цифрового решения может подойти;
* какие данные потребуются для запуска;
* какие существуют риски и ограничения;
* с чего начать первую рабочую версию проекта.

Это не калькулятор точной стоимости и не окончательное коммерческое предложение. Результат сервиса — предварительный структурированный «Паспорт автоматизации бизнеса».

---

## 2. Целевая аудитория

* малый бизнес;
* предприниматели;
* самозанятые;
* эксперты;
* руководители небольших команд;
* компании, в которых сотрудники выполняют повторяющиеся операции вручную;
* пользователи, которые понимают, что теряют время на рутину, но не знают, что именно можно автоматизировать.

## 3. Основной принцип

Клиент не должен выбирать технические инструменты.

Не использовать в клиентской части названия:

* n8n;
* Apps Script;
* Cursor;
* вайбкодинг;
* конкретные языки программирования;
* конкретные API и технические термины.

Клиент должен видеть только понятный бизнес-результат.

Вместо технических названий использовать формулировки:

* мини-сервис;
* автоматизация процесса;
* ИИ-ассистент;
* система аналитики и отчётности;
* цифровой помощник;
* решение для обработки данных;
* комплексное решение с интеграциями.

---

## 4. Формат проекта

Создать статический сайт, который работает без серверной части и может быть опубликован через GitHub Pages.

Использовать:

* HTML;
* CSS;
* JavaScript;
* без внешних библиотек;
* без платных API;
* без хранения секретных ключей;
* без регистрации пользователя.

Структура проекта:

business-automation-passport

## 5. Дизайн

Создать визуальный стиль современного делового досье или персонального диагностического документа.

Основная концепция: пользователь не просто проходит тест, а последовательно формирует собственный «Паспорт автоматизации бизнеса».

### Визуальное направление

* тёплый светлый фон, напоминающий качественную бумагу;
* графитовый или почти чёрный основной текст;
* ультрамариновый или насыщенный кобальтовый цвет как основной акцент;
* небольшой сигнальный акцент терракотового или кораллового цвета только для важных предупреждений и действий;
* тонкие линии, сетка, нумерация разделов и визуальные вкладки;
* элементы делового документа: номер диагностики, дата формирования, статус, отметки и аккуратные штампы;
* строгая выразительная типографика;
* минимальное количество скруглений;
* много свободного пространства;
* адаптивность для компьютера и смартфона;
* достаточный контраст текста;
* аккуратные функциональные анимации без декоративной перегрузки.

### Особенности интерфейса

* шаги диагностики оформить как разделы персонального досье;
* индикатор прогресса должен напоминать последовательное заполнение документа;
* итоговый результат оформить как отдельный «Паспорт автоматизации»;
* предусмотреть удобный вид результата для сохранения, скачивания и печати;
* кнопки должны быть заметными, но не выглядеть как стандартные неоновые SaaS-кнопки.

### Не использовать

* фиолетовые и сине-фиолетовые градиенты;
* неоновое свечение;
* glassmorphism;
* размытые цветные пятна на фоне;
* чрезмерно округлые карточки;
* типовой тёмный интерфейс AI-сервиса;
* переключение светлой и тёмной темы;
* декоративные элементы, не связанные с диагностикой и результатом.

  * другое;

### Шаг 2. Что происходит сейчас

## 8. Расчёты

Сервис должен рассчитывать:

### Текущие трудозатраты

Перевести время одного выполнения в часы и рассчитать примерные трудозатраты за месяц.

Использовать понятные допущения:

* несколько раз в день — 3 раза в день;
* ежедневно — 22 рабочих дня в месяц;
* несколько раз в неделю — 3 раза в неделю;
* еженедельно — 4 раза в месяц;
* несколько раз в месяц — 3 раза в месяц;
* ежемесячно — 1 раз в месяц.

Умножать результат на количество участвующих сотрудников, если это применимо.

### Потенциально освобождаемое время

Не обещать автоматизацию на 100%.

В зависимости от повторяемости и стандартизации процесса показывать ориентировочный диапазон освобождаемого времени:

* низкий потенциал — до 20%;
* средний потенциал — 20–50%;
* высокий потенциал — 50–75%.

Обязательно указывать, что это предварительная оценка.

### Потенциал автоматизации

Показывать:

* низкий;
* средний;
* высокий.

Не использовать формулировки:

* простая задача;
* сложная задача;
* лёгкая разработка;
* можно сделать быстро;
* дешёвое решение.

Клиент не должен видеть внутреннюю оценку трудоёмкости разработки.

## 9. Логика рекомендаций

В зависимости от ответов предложить один или несколько форматов решения.

### Мини-сервис

Предлагать, если:

* задача локальная;
* есть понятные входные данные;
* нужен один конкретный результат;
* пользователь хочет самостоятельно пользоваться инструментом.

### Автоматизация процесса

Предлагать, если:

* данные регулярно перемещаются между системами;
* сотрудники повторяют одинаковые действия;
* процесс выполняется по понятным правилам.

### ИИ-ассистент

Предлагать, если:

* нужно анализировать текст;
* готовить ответы;
* обрабатывать документы;
* составлять рекомендации;
* классифицировать информацию;
* при этом результат должен проверять человек.

### Аналитика и отчётность

Предлагать, если:

* данные находятся в таблицах;
* регулярно формируются отчёты;
* требуется сводная информация;
* есть повторяющиеся расчёты.

### Комплексное решение

Предлагать, если:

* участвует несколько систем;
* процесс состоит из нескольких этапов;
* требуется передача данных;
* есть разные роли пользователей;
* присутствуют ограничения по безопасности.

Не показывать клиенту конкретный технический стек.

## 10. Результат диагностики

После заполнения формы сформировать отдельный экран с результатом.

### Блок 2. Рекомендуемый формат решения

Показать понятное название решения и объяснить, почему оно подходит.

### Блок 3. Ожидаемый эффект

Например:

* сокращение ручного переноса данных;
* ускорение подготовки отчёта;
* уменьшение повторных операций;
* снижение количества ошибок;
* освобождение времени сотрудников;
* единая структура данных.

Не обещать гарантированную финансовую выгоду.

### Блок 4. Что потребуется для запуска

Сформировать персональный список:

* примеры исходных данных;
* примеры правильного результата;
* описание текущего процесса;
* список используемых систем;
* доступы или тестовые данные;
* правила обработки информации;
* ответственный сотрудник для проверки.

### Блок 5. Риски и ограничения

Показывать риски в зависимости от ответов:

* персональные данные требуют дополнительной защиты;
* исходные данные не структурированы;
* отсутствует единый источник информации;
* результат сильно зависит от экспертного решения человека;
* требуется обязательная ручная проверка;
* возможны ограничения используемых сервисов;
* перед разработкой необходимо проверить доступность интеграций.

### Блок 6. Черновик первой версии

Сформировать предварительный MVP:

* цель;
* что поступает на вход;
* что делает решение;
* что получает пользователь;
* что проверяет человек;
* какой результат считается успешным.
